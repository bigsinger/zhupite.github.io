2026 年 6 月，清华大学清新研究团队发布了 80 页的《2026 智能体安全研究报告》。报告原文可在 [book118](https://max.book118.com/html/2026/0607/7150136160011120.shtm) 查阅，搜狐有[发布报道](https://www.sohu.com/a/1034036233_468661)。

过去一周 Agent 安全密集发布——Unit 42 的 Skill 供应链验证、微软的评估框架、Dapr 的可信执行、AVP 的凭证代理——但清华这份报告是唯一试图做 **完整理论建构** 的。它的七层安全控制框架不是产品设计文档，而是一个参考架构。参考架构意味着它告诉你"应该有哪些组件"，但不告诉你每个组件怎么做——后者正是架构决策的职责。

本文不重复报告内容（那些原文已有），而是从 **架构决策** 的视角分析每层实现时必须面对的关键取舍。

---

## 一、身份层：三层链路的工程代价

报告要求每个操作记录 `{human_user → agent_instance → service_account}` 三元组。这听起来简单，但落地有两个根本性问题：

**问题 1：谁签发的身份？**

- **选项 A：SPIFFE/SPIRE（工作负载身份）** — Agent 进程启动时通过 SPIRE Agent 获取 SPIFFE ID，绑定到容器/进程元数据。优点是自动轮换、短 TTL，与 Kubernetes 天然集成。缺点是需要额外运维 SPIRE 集群，且 SPIFFE 主要用于服务间认证，不直接表达"人类用户"这个维度。
- **选项 B：OAuth 代理模式（如 AVP）** — Agent 通过本地代理获取短期 OAuth Token，凭证保存在外部 Vault，Agent 进程本身不持凭证字节。优点是人类用户维度可通过 Token Exchange 携带。缺点是每次工具调用增加一次 Token 获取的 RTT。

**问题 2：三层链路在 API 调用中怎么传递？**

如果下游服务不支持自定义 Header（很多 SaaS API 不认 `X-Agent-Identity`），三层链路只能在内部网关层终结，无法穿透到最终服务。这意味着审计日志只能由网关统一记录，不能依赖下游服务的审计。

> **架构决策小结**：推荐 **SPIFFE + OAuth 代理双轨**——服务间用 SPIFFE，外部 API 用 AVP 模式。三层链路在统一网关终结，不依赖下游支持。

## 二、权限层：最小权限的粒度博弈

报告提出"按任务分配权限，而非按角色"。这个原则是对的，但它带来了经典的 **权限粒度 vs 性能** 博弈：

**粗粒度（角色继承）**：Agent 继承人类用户的所有权限。优点是零改动、零延迟，当前大多数企业就是这么做的。缺点是 Agent 能做的事和人类一样多——人类用户有数据库写权限，Agent 也有。

**细粒度（按任务分配）**：每次任务动态计算最小权限集。优点是安全。缺点是：
- 权限计算有延迟（动态策略评估需要时间）
- 任务边界不清晰（"写一封周报邮件"需要哪些权限？发件箱写 + 文件系统读 + LLM 调用，还是更多？）
- 权限回收需要联动——任务完成不等于资源释放

**工程上可行的折中**：**权限标签预声明**。Agent 在注册时声明自己需要的权限范围（白名单），运行时不再扩大。这不是"按任务分配"，而是"按 Agent 分配最小权限"——比继承好，比动态计算简单。报告建议"优先落地 L2-L3"，我个人理解这正是其隐含策略：L2 对应白名单模式，L3 对应关键操作人工审批，动态权限可以先不做。

## 三、工具层：注册中心的设计分歧

报告提出"工具注册中心"作为集中管控点。但注册中心本身的架构有截然不同的选择：

**推模型（Agent 主动注册）**：Agent 启动时向注册中心报告自己的工具集。优点是灵活，Agent 可以动态加载新工具。缺点是 Agent 可能谎报——恶意 Agent 可以只注册部分工具，实际调用未注册的工具。

**拉模型（控制平面定期扫描）**：控制平面从可观测性数据中发现 Agent 实际调用的工具，与预期比对。优点是不可绕过——不管 Agent 是否诚实注册，调用痕迹都会暴露。缺点是滞后——攻击可能已经在注册扫描前完成了。

> **架构决策小结**：**推 + 拉双通道**是推荐模式。推用于"允许列表"（上线前审批），拉用于"监控报警"（发现未注册调用）。单靠任一都有盲区。

工具层的第二个架构问题是 **MCP Server 怎么管**。MCP Server 是 Agent 的工具入口，但 MCP 协议本身不包含安全元数据——它不告诉调用方"这个工具有什么风险级别"。报告的策略引擎需要在工具注册时人工标注风险标签，这意味着每个 MCP Server 上线前需要经过安全评审。这在小规模可行，但在 MCP 生态爆发时（成千上万个社区 MCP Server），评审速度会成为瓶颈。需要自动化工具风险评估（类似 app store 的自动化审核），但报告未深入讨论这一点。

## 四、上下文层与记忆层：有状态安全的新挑战

这两层放在一起讨论，因为它们解决的是同一个根本问题——**Agent 是有状态系统**，这是 Agent 安全与 Web 安全最大的结构性差异。

### 上下文层：指令与数据的分离

报告要求在系统层面区分"用户指令"和"不可信数据"，但实现这个区分在架构上非常棘手：

- **RAG 场景**：用户问"根据最新的财报数据写摘要"——这条指令里，"写摘要"是用户意图，"财报数据"是数据来源。但 LLM 本身不区分这两者。
- **多轮对话**：上一轮用户说"记住我的数据库密码是 xxx"，下一轮说"帮我连接数据库"——第一轮的信息应该视为"数据"还是"指令"？
- **工具输出**：Agent 调用搜索引擎返回的结果，里面可能包含恶意文字——但 Agent 必须读取工具输出来决定下一步。

目前最务实的做法是 **数据来源打标**：系统在上下文注入时给每条内容加上元标签（`[user_instruction]`、`[rag_source]`、`[tool_output]`），通过系统提示词让 LLM 感知这些标签。这不是完美方案（LLM 可能忽略标签），但它是无需修改模型即可落地的架构。

### 记忆层：版本化存储的代价

报告提出记忆需要版本管理和回滚，这在实际落地时面临一个核心矛盾：

**记忆的存储模型选择**：
- **向量库快照模式**：定期全量备份向量库，回滚时整个还原。优点是简单。缺点是粗粒度——回滚一个 Agent 可能影响其他共享同一向量库的 Agent；且向量库数据量大，快照开销高。
- **操作日志重放模式**（Event Sourcing）：记录每次记忆写入操作，回滚时逆向执行。优点是细粒度。缺点是重放时间长（一个长期运行的 Agent 可能有数十万次记忆操作），且"撤销"记忆操作可能违反业务约束（已发送的邮件无法撤回）。

> **架构决策小结**：推荐 **分层存储**——短期记忆用操作日志（支持细粒度回滚），长期记忆用向量库快照（支持粗粒度恢复）。两者配合可以覆盖大多数场景，但增加了一个存储层的运维复杂度。

报告将记忆污染列为独立风险类别，这是正确的——但它同时暗示了记忆版本化是解决之道。真实世界中，记忆版本化的工程复杂性远高于报告描述，企业应该先做"敏感信息不过期"（防止凭证记入记忆），再做"写后审计"（发现污染），最后才是"回滚"。

## 五、沙箱层：隔离粒度的折中

Agent 执行的沙箱化是行业共识，但隔离粒度直接影响 Agent 能做什么。报告建议了沙箱+网络限制+资源限制+快照回滚，但未讨论最关键的决策点——隔离粒度：

| 粒度 | 技术选型 | 隔离强度 | 适用场景 | 代价 |
|------|---------|---------|---------|------|
| **进程级** | WASM 沙箱 (extism/wasmtime) | 弱 | 纯计算、无 I/O | 低延迟，约 1-5ms |
| **容器级** | gVisor (runsc) | 中 | 文件操作、命令执行 | 启动 100-500ms，资源开销中等 |
| **VM 级** | Firecracker / Kata | 强 | 高风险操作、多租户隔离 | 启动 1-3s，资源开销高 |

对于一个通用 Agent 框架来说，选择哪一级隔离取决于 Agent 的具体能力。如果 Agent 只做 API 调用，WASM 足够了。如果 Agent 能执行任意代码（如 OpenAI Code Interpreter），需要容器级或 VM 级。

> **架构决策小结**：推荐 **分级沙箱**——根据工具风险标签动态选择隔离粒度。低风险工具走 WASM，中风险走容器，高风险走 VM。这需要沙箱调度器支持多种运行时，增加了实现复杂度，但避免了"一刀切"的性能浪费。

## 六、审计层：查询性与防篡改的矛盾

报告要求全量操作日志（Action Ledger），这是最基本的。但从架构上，审计系统的设计面临一组矛盾：

**存储模型的三条路径**：

1. **普通关系库 + RBAC 保护**：最简单的方案。但 DBA 可能有权限修改日志，防篡改性弱。
2. **Append-only 日志 + 加密校验链**：类似 Prometheus TSDB 或 Elasticsearch 的写入保护。写入后可追加不可修改。防篡改性强于方案 1，但数据库管理员仍有操作系统的文件级访问权限。
3. **区块链式链式哈希**：每条日志记录前一条的哈希。优点是篡改可检测。缺点是查询困难（你要查某条记录，需要从头 Hash 链验证）；存储膨胀（每条记录额外存哈希指针）。

**大多数企业的现实选择**：方案 2（加密校验的 Append-only）。方案 3 的查询代价在现代 DevOps 节奏下不可接受——当事故发生时，安全团队需要在 5 分钟内找到相关日志，而不是花 5 分钟验证 Hash 链。

审计层的另一个架构问题：**Agent 的审计日志和业务日志应该在同一管道吗？** 推荐**分离**——Agent 操作日志走独立管道（即使下游是同一个 Elasticsearch 集群），因为 Agent 的调用频次和写入模式与业务日志不同（高频、小负载、突增），混在一起可能影响业务日志的检索性能。

## 七、控制平面本身：一个独立基础设施

统一控制平面（注册中心 + 策略引擎 + 基础安全 + 审计应急）是报告架构的核心创新。它不是一个产品，而是一个独立的基础设施层——类比 Kubernetes 的控制平面 vs 数据平面。

架构上，这意味着：

**控制平面必须是高可用的**。如果策略引擎挂了，Agent 还能不能执行操作？理论上应该"拒绝所有调用"（fail close），但业务团队会反对——Agent 不能工作意味着业务停摆。折中是"降级为日志模式"（fail open 但记录），业务继续但安全团队会收到告警。

**控制平面的部署模型**：
- **Sidecar 模式**（如 Envoy/AVP）：每个 Agent 部署附带一个控制平面代理。延迟低，但更新所有 sidecar 是个运维问题。
- **中央网关模式**：所有 Agent 调用经过统一网关。运维简单，但网关成为单点瓶颈和延迟跳。
- **混合模式**：sidecar 做本地策略缓存 + 本地决策（低延迟），定期同步中央网关的全局策略。这是参考架构推荐的方向，但缓存一致性是个难题。

**策略引擎的运行时**：OPA (Open Policy Agent) 是目前最成熟的选择。但 OPA 的 Rego 语言主要面向传统策略场景（"用户是否有权限访问资源？"），Agent 安全的策略需要理解"这个工具调用的上下文是否安全"——这是一个更复杂的判断，可能需要结合实时威胁情报和异常检测模型。

## 结语：框架的价值在抽象层

清华报告最大的价值不是每一个具体的技术方案——那些会被产品迭代快速覆盖。它的价值在于 **给出了一个完整的抽象层**，让行业各方能在共同的框架下对话：

- AVP 在身份层 → "你的方案填补了身份链路的哪一段？"
- Unit 42 在工具层 → "你的验证方法属于签名验证还是行为基线？"
- 微软评测在评测层 → "你的测试覆盖了五维中的哪几个？"
- Dapr 在审计层 → "你的可信执行解决了 Action Ledger 的什么问题？"

如果没有这个抽象层，这些发布就是各自孤立的。有了它，每份发布都能在框架中找到自己的位置，行业也能更清晰地看到哪里已经覆盖、哪里仍是空白。

对于正在选型的企业，建议以报告的七层框架做 **覆盖率评估**——哪些层已有成熟方案（如身份层有 AVP、沙箱层有 gVisor/Firecracker），哪些层需要自己建设（如策略引擎、记忆版本化）。不要试图一次覆盖所有层，选 2-3 个风险最高的层优先落地。

**框架是免费的，落地才是成本。**
