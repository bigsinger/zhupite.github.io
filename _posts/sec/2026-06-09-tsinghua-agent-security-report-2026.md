2026 年 6 月，清华大学清新研究团队发布了 80 页的《2026 智能体安全研究报告》。报告在搜狐等平台发布，原文可在 [book118](https://max.book118.com/html/2026/0607/7150136160011120.shtm) 查阅。

这是过去一周 Agent 安全密集发布潮中唯一一份来自学术界的系统性研究成果。和其他所有发布不同——它不是产品白皮书，不是监管指南，不是标准草案——它是一个学术团队对这个新兴领域的完整理论建构。

---

## 一、核心定义：Agent Safety = 身份 + 策略 + 工具 + 日志

报告用一个简洁的公式给出了 Agent 安全的定义：

> **Agent Safety = 身份 + 策略 + 工具 + 日志**

这个公式背后的逻辑链条是：

**身份**解决"谁在行动"的问题。Agent 需要独立的身份链路，不能共享人类用户的身份。每次操作需要完整记录执行人（人类用户）、Agent（智能体实例）、服务账号（后端身份）三层信息。这是可追溯性的前提。

**策略**解决"能做什么"的问题。策略必须在系统层执行，而非依赖提示词。每次工具调用前校验身份、权限、数据，遵循最小权限原则，按任务分配短期可撤销权限。

**工具**解决"操作什么"的问题。每个工具需要标注风险标签与调用规则，调用前后做策略校验。工具风险等级决定了 Agent 使用它时需要的审批层级。

**日志**解决"如何追溯"的问题。全量操作日志（Action Ledger）记录每次调用的输入、输出、决策结果，高风险操作设置人工审批门，实现全程可追溯。

报告同时给出两个关键的定性判断：
- Agent 不是聊天机器人，是**带权限的运行时系统**——安全边界远大于对话模型
- **企业核心竞争力在于安全控制平面**——模型可以换，控制平面是长期资产

## 二、七层安全控制框架的深入技术分析

报告提出了一个从数据接收到底层执行的全链路安全控制框架，共七层：

### 第一层：身份层（Identity Layer）

身份层解决的根本问题是**操作归属**。报告指出当前企业最常见的安全漏洞是"Agent 共享人类身份"——Agent 使用的人类 OAuth Token、Session Cookie 或 API Key 通常具有远超 Agent 实际需要的权限。

身份层的技术要点：
- **三层身份链路**：每个操作记录 `{human_user → agent_instance → service_account}` 三元组
- **禁止长期持有人类凭证**：Agent 必须使用独立的服务身份
- **与 IAM/IdP 系统集成**：身份声明应来源于企业的统一身份平台

这与 AVP（Agent Vault Proxy）的思路形成直接呼应——AVP 解决的是"Agent 进程不持有凭证字节"，身份层解决的是"Agent 的身份凭证来源可审计"。

### 第二层：权限层（Permission Layer）

权限层的核心是**运行时最小权限**。报告特别批评了"让 Agent 继承用户全量权限"的做法——这是目前最普遍也最危险的企业实践。

权限层技术设计：
- **按任务分配**——不是按角色分配，而是按具体任务分配最小权限集
- **短期可撤销**——权限有 TTL，任务完成后自动回收
- **分级操作模式**：
  - 只读操作：可自动执行
  - 半自动操作：Agent 提议，人工确认后执行
  - 自动执行：在严格定义的规则内自动执行
- **高风险操作强制审批**：资金操作、生产变更等必须有人工审批门

### 第三层：工具层（Tool Layer）

工具层承担着**能力出口**的管控职责。Agent 的能力最终通过工具调用体现，因此工具层是安全控制的关键卡口。

技术设计：
- **工具风险标签**：每个工具注册时标注风险等级（低/中/高）
- **调用前校验**：策略引擎在每次工具调用前检查身份、权限、上下文是否匹配
- **调用后审计**：工具调用的输入输出记录到审计日志
- **工具注册中心**：全量登记工具信息，未登记的工具不可被 Agent 调用

这与 Unit 42 的 Skill 供应链验证、JFrog 的 IDE 插件形成互补——工具层关注的是"工具调用时的安全"，Unit 42 关注的是"工具来源的可信"，JFrog 关注的是"工具开发过程中的安全"。

### 第四层：上下文层（Context Layer）

上下文层解决的是**输入可信度**问题。Agent 在交互中接收来自多个渠道的数据——用户指令、外部文档、API 响应、工具输出。这些来源的数据可信度不同，需要分层隔离。

技术要点：
- **数据来源标记**：区分"用户意图"、"可信数据"、"不可信数据"
- **上下文隔离**：不可信数据不能影响 Agent 的安全决策
- **指令与数据分离**：系统指令与用户提供的数据应严格区分

这是对抗**间接提示注入**的关键防线。当 Agent 读取的网页内容或邮件中包含恶意指令时，上下文层应该将其标记为"不可信数据"而非"用户指令"。

### 第五层：记忆层（Memory Layer）

记忆层解决的是 Agent 安全特有的一个威胁——**跨任务风险传递**。传统安全中每个请求是独立的，但 Agent 有长期记忆，上一次交互可能影响下一次交互的安全性。

技术要点：
- **记忆读写的权限管控**：不是所有 Agent 都能读写所有记忆
- **版本回溯**：支持记忆的版本管理和回滚
- **删除与回滚**：被污染的记忆可被隔离或删除
- **敏感信息过滤**：凭证、密钥不应写入长期记忆

这种持续性风险在传统安全中很少见。一个被污染的 Agent 可能在一周后仍然受之前恶意输入的影响。报告将其识别为独立的威胁类别，这是一个关键的理论贡献。

### 第六层：沙箱层（Sandbox Layer）

沙箱层提供**执行环境隔离**。代码执行、文件操作、命令执行等高风险操作必须在受限环境中进行。

技术要点：
- **环境隔离**：代码执行在独立的沙箱容器中运行
- **网络限制**：沙箱内运行的操作只能访问白名单内的网络资源
- **资源限制**：CPU、内存、磁盘写入量受限
- **快照与回滚**：沙箱操作可回滚到执行前的状态

这与 OpenAI 将沙箱执行作为 Agent 基础设施的策略一致。报告明确指出，执行层安全已成为 Agent 的标配能力。

### 第七层：审计与人工层（Audit & Human Layer）

这是最上层的兜底机制。当所有自动化安全机制都没能拦截到恶意行为时，审计日志提供追溯能力，人工审批提供最终防线。

技术要点：
- **Action Ledger**：全量操作日志，记录每次调用的完整链路信息
- **熔断机制**：检测到异常行为模式时自动暂停 Agent 操作
- **故障回滚**：支持将 Agent 的状态回滚到安全时间点
- **专属事故响应剧本**：预定义的 Agent 安全事件响应流程

## 三、十大高频风险的分类体系

报告将风险分为四个维度、十大类，比传统安全分类多了一个关键的维度——持续性风险：

### 主动攻击类（利用性）

| 风险 | 技术根源 | 影响 |
|------|---------|------|
| **目标劫持** | Agent 的规划能力被利用，任务目标被篡改 | Agent 执行攻击者意图 |
| **提示词注入** | 输入数据与系统指令未分离 | Agent 执行非授权操作 |
| **工具滥用** | 工具调用前缺乏策略校验 | Agent 越权调用工具 |

### 持续类（跨任务传递）

| 风险 | 技术根源 | 影响 |
|------|---------|------|
| **记忆污染** | 记忆层缺乏写管控和版本回溯 | 风险跨 session 延续 |
| **上下文投毒** | 不可信数据与指令未隔离 | Agent 在后续交互中被持续影响 |

这是 Agent 安全独有的威胁类型。传统 Web 安全中，每个 HTTP 请求是独立的；但 Agent 有状态、有记忆，攻击者可以通过一次注入影响后续所有交互。

### 扩散类（范围放大）

| 风险 | 技术根源 | 影响 |
|------|---------|------|
| **沙箱逃逸** | 执行环境隔离不足 | 攻击者获得宿主机访问权限 |
| **供应链污染** | 工具/Skill 的来源未验证 | 恶意代码通过合法渠道分发 |
| **多 Agent 级联失败** | Agent 间无安全信任边界 | 一个 Agent 沦陷导致连锁反应 |

### 衍生类（副效应）

包含数据泄露、意外代码执行、人机信任滥用、行为漂移、审计缺失等。这些不是独立的攻击向量，而是前述风险的副效应。

## 四、统一控制平面架构

报告提出，企业规模化治理 Agent 的核心不在于单个安全产品，而在于一个**统一控制平面**。控制平面包含四个核心模块：

### 模块一：Agent / 工具注册中心

所有 Agent 和工具在上线前必须在注册中心登记，标注：
- Agent 的身份标识、发布者、版本
- 工具的 API 端点、风险等级、预期调用参数
- 使用该工具的授权 Agent 列表

**未登记的主体禁止访问业务系统。** 这是一个关键的"默认拒绝"设计。

### 模块二：策略引擎（Policy Engine）

策略引擎是控制平面的决策核心。每次工具调用前执行以下检查链：

```
Agent 发起工具调用请求
        │
        ▼
策略引擎检查链：
  1. 身份校验：Agent 身份是否有效？执行人是否有权限？
  2. 权限校验：此操作是否在 Agent 的授权范围内？
  3. 数据校验：输入参数是否符合预期格式和范围？
  4. 上下文校验：当前上下文是否支持此操作？
  5. 风险评分：综合以上维度给出风险评分
        │
        ▼
决策结果：允许 / 拒绝 / 降级（转为人工审批）
```

策略引擎不是在提示词层面工作，而是在**系统调用层面**工作——它拦截的是实际的工具调用，而不是 LLM 的输出。这意味着即使提示词被绕过，策略引擎仍然能拦截越权操作。

### 模块三：基础安全能力

- **上下文防火墙**：过滤输入中的恶意内容，标记不可信数据
- **数据脱敏 + DLP**：对 Agent 的输入输出做敏感数据检测和脱敏
- **沙箱隔离**：高风险操作在受限环境中执行
- **分级人工审批**：根据不同风险等级设置不同审批流程

### 模块四：审计与应急

- **全链路操作日志**：记录从请求到执行的完整链路
- **熔断机制**：检测到异常行为模式时自动暂停
- **故障回滚**：支持 Agent 状态回滚
- **专属事故响应剧本**：预定义的 Agent 安全事件响应流程

## 五、评测体系：五维测试 + 常态化监控

报告指出，Agent 安全评测不是一次性测试，而是持续运营能力。评测体系分为五个核心维度：

**1. 注入攻击红队测试**
自动化生成对抗性输入（直接注入、间接注入、多轮递进注入），测试 Agent 的防御能力。核心指标包括注入成功率、拒绝率、敏感信息泄露率。

**2. 工具调用安全测试**
验证 Agent 在授权范围内调用工具的行为是否符合预期。测试 Agent 是否会在授权范围内执行越权操作（如使用只读工具的 delete 参数）。

**3. 数据泄露检测**
检查 Agent 输出是否包含 PII、凭证、内部信息等敏感数据。

**4. 记忆污染验证**
测试 Agent 在对抗性输入后是否会将恶意内容写入长期记忆，以及被污染的记忆是否会影响后续操作的安全性。

**5. 沙箱逃逸对抗测试**
模拟攻击者尝试从沙箱环境中逃逸的攻击路径。

运行时的监控需要对接 SIEM 等安全平台，监控 Agent 的调用频率、行为模式、异常行为，触发自动降级或暂停。

## 六、治理体系：三级责任 + 五分成熟度

### 三类负责人

报告创新性地提出每个 Agent 配备三位负责人：
- **业务负责人**：定义 Agent 的业务目标和范围
- **技术负责人**：负责 Agent 的技术实现和运维
- **安全负责人**：负责 Agent 的安全策略和审计

三位负责人的权责清晰分离——这在企业级落地时能有效防止"业务赶进度、安全被忽略"的问题。

### 五级能力成熟度模型

| 等级 | 名称 | 核心特征 | 管控程度 |
|------|------|---------|---------|
| **L1** | 只读辅助 | Agent 只能读取信息，不可执行任何业务操作 | 最严格 |
| **L2** | 受限工具 | Agent 可使用有限工具集执行受约束操作 | 严格 |
| **L3** | 半自动 | Agent 可执行大部分操作，高风险需人工审批 | 适中 |
| **L4** | 闭环自治 | Agent 在业务规则内全自动执行，无需人工介入 | 宽松 |
| **L5** | 高保证自治 | Agent 完全自治，具备自愈和自适应安全能力 | 自治 |

报告的务实之处在于建议**优先落地 L2-L3**。不是追求最高的安全等级，而是在"Agent 的效率提升"和"安全风险可控"之间找到平衡点。L2 意味着 Agent 能做具体的事情但受限，L3 意味着 Agent 能高效执行但保留人工审批。

### 风险分级落地策略

| 风险等级 | 典型场景 | 管控措施 |
|---------|---------|---------|
| **高风险** | 资金操作、医疗决策、法律判断、生产变更、身份权限操作 | 强制人工审批 + 全链路审计 |
| **中风险** | 客户外发、公开发布 | 审核后执行 |
| **低风险** | 普通内容交互、知识检索、报告撰写、代码辅助 | 可直接自动化 |

建议优先落地低风险、可回滚任务。

## 七、落地路线图：180 天到一年

| 时间窗口 | 核心任务 | 产出 |
|---------|---------|------|
| **0-30 天** | 资产盘点：梳理所有 Agent 与高权限工具 | 完整的 Agent/工具资产清单 |
| **30-60 天** | 搭建基线：权限模型、日志体系、审批流程 | 统一安全策略文档 |
| **60-90 天** | 试点验证：选取低风险场景试点 + 红队渗透测试 | 试点评估报告 |
| **90-180 天** | 平台化建设：注册中心、策略引擎、沙箱基础设施 | 统一控制平面 V1 |
| **一年目标** | 核心 Agent 全面接入控制平面，高风险场景专项评估 | 企业级 Agent 安全体系 |

## 八、报告的三个关键洞察

### 洞察一：从模型安全到运行时安全

报告明确指出 Agent 安全的主战场已从模型安全转向运行时系统安全。这不是说模型安全不重要——而是说模型安全只是 Agent 安全的一部分。Agent 的运行时系统（工具、身份、沙箱、审计）构成了更大的攻击面。

### 洞察二：抽象层高于实现层

报告提出了一个完整的抽象层（七层框架、控制平面、风险分类、成熟度模型）——这个抽象层不绑定任何具体技术、产品和平台。这意味着它不会被任何单一厂商锁定。今天可以用 OpenAI Agent，明天可以切换 Anthropic——控制平面是独立于 Agent 实现的。

### 洞察三：安全是持续运营不是一次性检查

报告反复强调，Agent 安全不是上线前的一次安全评估，而是持续运营能力。评测需要常态化、监控需要 7×24、事故响应需要预演。这是"安全不是可选项"的真实含义——不是让你买一个安全产品，而是让你建设一个安全体系。

## 结语

清华这份报告的核心价值不是告诉我们"Agent 安全很重要"——这个结论行业已经形成共识了。它的真正贡献是给出了一个**系统化的理论框架**，把过去一周碎片化的行业发布串了起来。AVP 在身份层、Unit 42 在工具层和控制平面、Dapr 在审计层、微软评测框架在评测层、五眼联盟指南在治理层——所有这些都能在报告的七层框架中找到自己的位置。

对于正在评估 Agent 安全的企业，建议按报告给出的路线图行动：**先做资产盘点，再建安全基线，然后试点验证，最后平台化。** 这个路线图的务实之处在于它没有跳步——不要求你第一天就建好控制平面，而是让你先用 30 天搞清楚自己有多少 Agent 和工具再说。
