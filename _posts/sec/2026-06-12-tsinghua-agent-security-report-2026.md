6 月 12 日，清华大学清新研究团队发布了 80 页的《2026 智能体安全研究报告》。

这份报告和过去一周所有行业发布都不一样。它不是某个安全厂商的产品方案，不是某个标准组织的指南，不是开源社区的工具——它是一个学术团队对 Agent 安全这个新兴领域的**系统化理论建构**。

报告的主要发现和建议通过搜狐等媒体发布，全文可在 [book118](https://max.book118.com/html/2026/0607/7150136160011120.shtm) 获取。

---

## 一个公式定义 Agent 安全

报告开门见山给出了一个非常简洁的定义：

> **Agent Safety = 身份 + 策略 + 工具 + 日志**

这个公式说清楚了四件事：
- **身份**：Agent 是谁？谁在授权它行动？
- **策略**：Agent 能做什么、不能做什么？
- **工具**：Agent 调用的外部系统是否可控？
- **日志**：Agent 做过什么可以被追溯？

四个要素缺一不可。没有身份，Agent 和攻击者的请求无法区分；没有策略，Agent 的权限不受约束；没有工具安全，Agent 的能力不可控；没有日志，Agent 的行为不可审计。

## 三个核心判断

报告基于大量分析，给出了三个判断：

**判断一：Agent 不是聊天机器人，是带权限的运行时系统。**

这意味着 Agent 的安全边界远大于传统对话模型。一个聊天机器人的风险止于"说了不该说的话"，而一个 Agent 的风险可以延伸到"做了不该做的事"——数据篡改、业务变更、资金操作等实际损失。

**判断二：仅靠提示词无法保障工具调用与外部动作的安全。**

提示词可以被绕过、注入、忽略。把 Agent 的安全寄托在 prompt 工程上，相当于用防火墙的策略只写在 README 里。安全必须在系统层执行。

**判断三：企业的核心竞争力在于安全控制平面，而非单个大模型。**

模型可以换、API 可以切、Agent 框架可以升级——但安全控制平面（身份、权限、审计、策略引擎）一旦搭建，就是企业的核心安全基础设施。模型在快速商品化，安全部署能力无法自动复刻。

## 七层安全控制框架

这是报告最核心的贡献——一个从底层到顶层的全链路控制框架：

| 层级 | 职责 | 具体措施 |
|------|------|---------|
| **身份层** | 谁在使用 Agent？ | 配置独立身份链路，禁止长期持有人类凭证，完整记录执行人、Agent、服务账号信息 |
| **权限层** | Agent 能做什么？ | 最小权限，按任务分配短期可撤销权限，区分只读/半自动/自动执行，高风险动作强制审批 |
| **工具层** | Agent 调用的工具是否安全？ | 工具标注风险标签与调用规则，调用前后做策略校验 |
| **上下文层** | Agent 的输入是否可信？ | 隔离不可信数据，区分指令、用户意图与普通数据，防范上下文投毒 |
| **记忆层** | Agent 的长期记忆是否被污染？ | 管控记忆读写，支持版本回溯、删除与回滚 |
| **沙箱层** | Agent 的执行环境是否隔离？ | 代码、文件、命令操作强制环境隔离，限制网络与资源，支持快照与回滚 |
| **审计&人工层** | 出事了能否追溯？ | 全量操作日志（Action Ledger），高风险不可逆动作设置人工审批门 |

这个框架的一个显著特点是：**它不是从攻击者的视角（MITRE ATT&CK）出发，而是从防御者的工程落地视角出发**。每一层都对应着明确的工程动作——建身份、配权限、打标签、做隔离、写日志。

## 十大高频风险

报告将 Agent 面临的风险分为四类十种：

**攻击类（主动利用）：**
- 目标劫持、提示词注入、工具滥用

**持续类（跨任务延续）：**
- 记忆污染、上下文投毒

**扩散类（影响放大）：**
- 沙箱逃逸、供应链污染、多 Agent 级联失败

**衍生类（副效应）：**
- 数据泄露、意外代码执行、人机信任滥用、行为漂移、审计缺失

特别值得注意的是**记忆污染和上下文投毒**被列为持续性风险。这意味着 Agent 的上一次交互可能影响下一次交互的安全性——这在传统安全中很少见，是 Agent 安全独有的威胁模型。

## 风险分级与落地策略

报告按风险等级分类：

| 风险等级 | 典型场景 | 建议策略 |
|---------|---------|---------|
| **高风险** | 资金操作、医疗决策、法律判断、生产变更、身份权限操作 | 强制人工审批 + 全链路审计 |
| **中风险** | 客户外发、公开发布 | 审核后执行 |
| **低风险** | 普通内容交互、知识检索 | 可直接自动化 |

建议：低风险任务先行落地，逐步向中高风险扩展。

## 统一控制平面

企业规模化治理 Agent 的核心是统一控制平面，需要建设四个核心模块：

1. **Agent/工具注册中心**——全量登记智能体和工具，标注风险等级，未登记主体禁止访问业务系统
2. **策略引擎**——系统层执行安全规则，每次工具调用前校验身份、权限、数据
3. **基础安全能力**——上下文防火墙、数据脱敏 + DLP、沙箱隔离、分级人工审批
4. **审计与应急**——全链路操作日志、熔断机制、故障回滚、专属事故响应剧本

## 评测体系：不是一次性测试，是持续运营

报告提出的评测框架覆盖五个维度：

- 注入攻击红队测试
- 工具调用安全测试
- 数据泄露检测
- 记忆污染验证
- 沙箱逃逸对抗测试

以及运行监控能力：对接 SIEM 等安全平台，监控调用频率、异常行为，触发自动降级或暂停。事故响应遵循"检测-遏制-根除-恢复"流程。

## 五级能力成熟度模型

| 等级 | 名称 | 特征 |
|------|------|------|
| **L1** | 只读 | Agent 只能读取信息，不可执行业务操作 |
| **L2** | 受限工具 | Agent 可使用有限工具集，执行受约束的操作 |
| **L3** | 半自动 | Agent 可执行大部分操作，高风险需人工审批 |
| **L4** | 闭环自治 | Agent 在业务规则内全自动执行 |
| **L5** | 高保证自治 | Agent 完全自治，具备自愈和自适应安全能力 |

建议企业优先落地 **L2-L3** 阶段——既获得了 Agent 的效率提升，又通过半自动模式保留了安全控制。

## 落地路线图

| 时间 | 任务 |
|------|------|
| **0-30 天** | 资产盘点：梳理所有 Agent 与高权限工具，建立基础清单 |
| **30-60 天** | 搭建基线：权限、日志、审批基线，制定统一安全策略 |
| **60-90 天** | 试点验证：选取低风险场景试点，完成红队渗透测试 |
| **90-180 天** | 平台化：完善注册中心、策略引擎、沙箱等基础设施 |
| **一年目标** | 建成企业统一安全控制平面，核心 Agent 全面接入 |

## 三方责任体系

每个 Agent 配备三类负责人：

- **业务负责人**——Agent 的业务目标和范围
- **技术负责人**——Agent 的技术实现和运维
- **安全负责人**——Agent 的安全策略和审计

权责清晰，这是企业级落地的重要组织保障。

## 报告中的三大企业误区

报告特别指出了当前企业在 Agent 安全上的三个常见错误认识：

1. **仅做提示词防护**——以为写了 system prompt 就安全了
2. **让 Agent 继承用户全量权限**——Agent 拥有什么权限取决于用户登录后能看到什么，这在授权上过于宽松
3. **先对接业务再补审计审批**——安全后置，导致上线后很难追加管控

这三点非常实际——几乎每个刚开始落地 Agent 的企业都会至少踩中一个。

## 结语

这份报告是过去一周 Agent 安全发布潮中唯一一个**学术视角的系统化建构**。它没有推销任何产品，没有绑定任何平台，只是客观地给出了威胁模型、控制框架、评测方法和落地路线。

对于正在评估 Agent 安全的企业来说，这份报告最有价值的部分不是理论框架，而是那个 0-30-60-90-180 天的落地路线图——它告诉你在每个阶段应该做什么，先做什么后做什么，以及一年后应该达到什么状态。这比任何安全产品都更值得先读。
