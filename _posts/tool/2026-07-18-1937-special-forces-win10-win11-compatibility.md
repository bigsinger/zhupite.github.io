## 1. 最终方案介绍

这不是一个单一的“兼容模式”问题，而是四条老式 Windows 游戏链路叠加造成的：

1. 原生 DirectDraw 显示路径与现代 Windows 不兼容；
2. 绿色版的资源库检查在当前环境下产生误报；
3. 启动影片和 DirectInput 轮询期间长期不处理 Windows 窗口消息，导致标题栏显示“未响应”；
4. 未同步的 60 FPS 提交与 Windows 桌面合成器相位漂移，造成肉眼可见的偶发重复帧/微卡。

最终方案由两层组成：

- 使用官方 [cnc-ddraw](https://github.com/FunkyFr3sh/cnc-ddraw) 将旧 DirectDraw 调用转换到 Direct3D 9，并负责窗口化、缩放和限速；
- 使用本游戏专用的 `dinput.dll` 兼容层转发系统 DirectInput、补充窗口消息处理，并在内存中跳过误报分支和两个问题启动影片。

磁盘上的原版 `M1937.exe` 没有被修改。最终又为图形层开启 VSync，使测试中的桌面合成丢帧率由 **18.81% 降为 0%**。

最终方案不是单纯打开 Windows“兼容模式”，而是在游戏进程外围补齐旧引擎依赖的运行环境：

| 层次 | 最终处理 | 解决的问题 |
|---|---|---|
| 图形兼容 | cnc-ddraw 7.1.0.0、Direct3D 9、固定 60 FPS、VSync | DirectDraw 报错、严重卡顿、桌面合成丢帧 |
| 输入与消息 | 本游戏专用 32 位 `dinput.dll` 代理 | 菜单、任务简报、关卡循环长期不处理窗口消息 |
| 游戏专用修复 | 校验版本和原字节后进行内存补丁 | 资源库误报和两个问题启动影片 |
| 显示模式 | 窗口启动、`Alt+Enter` 动态采用当前桌面分辨率 | Win10/Win11 全屏分辨率和缩放问题 |

所有游戏专用机器码修复都只存在于当前进程内存中。关闭游戏后即消失，原版 EXE、资源、关卡和存档格式均保持不变。补丁也不依赖后台服务或网络连接，可以跟随绿色版目录离线分发。

## 3. 故障排查过程

### 3.1 `DirectDraw Error DDERR_UNSUPPORTED`

原始报错位置：

```text
DirectDraw Error DDERR_UNSUPPORTED
FILE: E:\IntuitionEngine\IEngineGraphic\IEngineScreen.cpp
LINE: 521
```

游戏依赖 Windows XP 时代的 DirectDraw 表面、显示模式和锁定行为。在 Win10/Win11 上直接调用时，部分操作会返回 `DDERR_UNSUPPORTED`；即使偶尔成功，显示和限速路径也会严重卡顿。

处理方法：把官方 cnc-ddraw 的 `ddraw.dll` 放到游戏目录，使游戏的 DirectDraw 调用在进程内转换为现代渲染后端。最终选择 `renderer=direct3d9`，原因是这台机器上它能够稳定创建窗口、正确显示 2D 画面，并且 Present 调用耗时很低。

结果：`DDERR_UNSUPPORTED` 消失，窗口画面正常出现。

### 3.2 `Can't Load Game Files Library!`

该弹框并非资源文件真的丢失。排查时确认：

- `1937Resources.GFL` 和 `InterMedia.GFL` 均存在且能成功打开；
- 资源归档读取和解包流程已经通过；
- 弹框来自绿色版自身的旧环境检查/返回值判断，是误报。

没有采用“每次自动点击弹框”的临时方案，而是在启动时仅修改进程内存中的条件跳转：

- RVA：`0x0000734A`；
- 预期原字节：`74 0C`；
- 运行时替换：`EB 0C`。

补丁带有原字节校验；程序版本不匹配时不会盲目写入。

结果：弹框从根因处被抑制，游戏资源仍由原流程加载。

### 3.3 启动动画长时间“未响应”

两个启动影片为：

- `GamekingLogo.svt`；
- `1937Intro.svt`。

老影片播放/等待循环在当前系统上耗时很长，且期间不取出 Windows 消息队列中的消息，因此系统把窗口标为“未响应”。继续等待后可能进入菜单，但等待时间和体验不可接受。

处理方法：在内存中将只负责排入上述两个启动影片的 29 字节调用序列改为 NOP：

- RVA：`0x0000762C`；
- 长度：29 字节；
- 同样先校验完整原字节，再执行内存修改。

此修改只跳过片头，不删除 `.svt` 文件，也不跳过任务/关卡视频。

结果：启动直接进入菜单路径，不再经历片头假死。

### 3.4 点击“开始游戏”及任务简报阶段“未响应”

进一步测试发现，问题不只出现在片头：菜单、任务简报和游戏循环中的某些 DirectInput 轮询也会长时间占住线程，却不分发窗口消息。

专用 `dinput.dll` 的处理方式：

1. 从 Windows 系统目录加载真正的 `dinput.dll`；
2. 转发 `DirectInputCreateA` 及原 DirectInput 设备接口；
3. 在 `Acquire`、`GetDeviceState`、`GetDeviceData` 前处理最多 64 条待处理窗口消息；
4. 保留 `WM_QUIT`，正常分发键盘、鼠标、绘制和窗口状态消息。

这样没有改变游戏输入语义，只补上现代 Windows 判断窗口是否响应所需的消息泵。

结果：启动、菜单、任务简报和第一关地图阶段均持续 `Responding=True`。

### 3.5 可玩但偶尔“顿一下”

#### 排查假设

本机环境有几个潜在因素：

- 游戏位于 `ST1000DM003` SATA 机械硬盘；
- 同时存在 60 Hz、144 Hz 和虚拟显示适配器；
- 当前配置同时限制显示帧率和游戏逻辑 Tick；
- `vsync=false` 时，固定 60 FPS 与 60 Hz 桌面合成器可能发生相位漂移。

为避免截图采样自身影响游戏，使用 Intel 官方 [PresentMon](https://github.com/GameTechDev/PresentMon) 1.10.0 直接采集 Direct3D 9 Present 事件，并由自动测试进入第一关。

#### 原配置结果

原配置为 `maxfps=60`、`maxgameticks=60`、`limiter_type=4`、`vsync=false`。

第一关稳定阶段的 11 秒样本：

| 指标 | 结果 |
|---|---:|
| Present 数量 | 638 |
| 帧间隔 P50 | 16.672 ms |
| 帧间隔 P95 | 17.235 ms |
| 帧间隔 P99 | 17.409 ms |
| 最大帧间隔 | 17.856 ms |
| 超过 25 ms | 0 |
| 被桌面合成器丢弃 | 120（18.81%） |

这说明游戏和显卡有能力稳定地产生约 60 FPS，卡顿不是 GPU 性能不足，也不是这一段中机械硬盘阻塞了渲染；问题在于部分已提交帧没有被显示，玩家会看到间歇性的重复帧。

#### 方案对比

- `vsync=true`：丢弃帧降为 0，保留固定 60 FPS；
- `maxfps=-1`（自动跟随屏幕刷新率）：在本机多显示器/虚拟显示环境中误判，后段退化到约 10 FPS，已否决；
- 切换 OpenGL 或降低 D3D9 滤镜：现有 Present/GPU 开销已经很低，没有必要增加兼容性变量；
- 强制单 CPU 核：cnc-ddraw 自身也提示可能影响性能或声音，当前无冻结且单核未满载，因此保持关闭。

#### 最终长时间回归

最终配置使用固定 60 FPS + VSync。第一关连续 21.52 秒有效样本：

| 指标 | 结果 |
|---|---:|
| Present 数量 | 1292 |
| 帧间隔 P50 | 16.667 ms |
| 帧间隔 P95 | 17.100 ms |
| 帧间隔 P99 | 17.458 ms |
| 最大帧间隔 | 18.977 ms |
| 超过 25/33/50 ms | 0 / 0 / 0 |
| 被桌面合成器丢弃 | 0 |
| “未响应”采样 | 0 |
| 单个逻辑核心占用 | 约 21.1% |
| 关卡阶段进程读取量 | 约 167 KB |

因此，本次发现的偶发微卡已通过 VSync 修正。VSync 理论上会增加最多约一帧输入等待，但对这款即时战术游戏影响很小，换来的是稳定的显示节奏。

## 4. 最终核心配置

`ddraw.ini` 中对本游戏起关键作用的全局项如下：

```ini
[ddraw]
windowed=true
maintas=true
renderer=direct3d9
maxfps=60
vsync=true
adjmouse=true
noactivateapp=true
maxgameticks=60
limiter_type=4
singlecpu=false
toggle_upscaled=true
```

说明：

- `maxfps=60` 只限制实际显示提交频率，不改变游戏速度；
- `maxgameticks=60` 限制旧引擎逻辑循环，避免忙等占满一个核心；
- `limiter_type=4` 在 `PeekMessage` 路径限速，与本游戏消息循环最匹配；
- `vsync=true` 消除本机实测到的 DWM 合成丢帧；
- `singlecpu=false` 避免旧游戏被不必要地锁到一个核心；
- `toggle_upscaled=true` 使 `Alt+Enter` 全屏时采用当前显示器的桌面分辨率；本机当前为 1920×1080，系统分辨率改变后无需重新写死宽高；
- 窗口宽高和位置会由 cnc-ddraw 根据用户调整自动保存，不属于补丁关键参数。

### 4.1 全屏分辨率验证

全屏分辨率没有写死为 `width=1920`、`height=1080`，而是由 cnc-ddraw 在切换时读取当前显示器模式。这样以后把 Windows 桌面改成其他分辨率，游戏全屏也会自动跟随。

本机实测：

| 阶段 | 客户区尺寸 | 窗口位置 | 结果 |
|---|---:|---:|---|
| 切换前窗口模式 | 1024×768 | (440, 125) | 正常 |
| `Alt+Enter` 全屏 | 1920×1080 | (0, 0) | 与主屏当前分辨率一致 |
| 再次切回窗口 | 1024×768 | (440, 125) | 原尺寸恢复 |

测试期间进程始终保持响应。`maintas=true` 会保留游戏原始 4:3 比例，因此在 16:9 的 1920×1080 屏幕上左右留黑边，避免画面被横向拉伸；这是预期行为。

## 8. 后续使用注意事项

- 建议把游戏窗口留在当前 60 Hz 主显示器上。固定 60 FPS 在 144 Hz 屏幕上不能整除刷新周期，拖到另一块高刷屏后可能重新出现轻微节奏不均；
- 机械硬盘会让首次进入大关卡或读取大型资源时产生正常的加载等待。最终关卡稳定阶段并未发现磁盘读取引起的渲染停顿；若只在切换关卡时顿一下，可将整个绿色版复制到 SSD，而不是只移动部分资源；
- 不建议为游戏目录设置杀毒软件永久排除项。若第三方安全软件确实在每次读取旧资源时扫描，可先做一次按需扫描并观察，不应直接降低系统安全策略；
- 不要把 `maxfps` 改成 `-1`。本机有多个刷新率和虚拟显示适配器，实测自动刷新率模式会误判；
- 如果手动修改过 `ddraw.ini` 后问题复现，优先恢复本文第 4 节的十个关键参数。
